# Wiki Ingest Skill（v5）

## Goal

将 `raw/` 中的各类原始资料编译为结构化知识稿。摄入质量决定知识库的可信度——宁可标记"待确认"也不写入未经验证的内容。

## Read First

执行前必须读取：
1. `CLAUDE.md`
2. `schema/lint-rules.md`
3. `wiki/index.md`
4. `wiki/log.md`
5. `.agent-cache/cache/ingest-manifest.json`（如不存在则跳过）

---

## Extraction Rules（提取规则）

按 `raw/` 原始资料的文件类型使用对应专用能力提取内容，**禁止把临时脚本当作首选提取方式**：

| 文件类型 | 提取方式 | 注意事项 |
|---|---|---|
| `.md` | Read 工具直接读取 | 唯一可直接读取的结构化格式 |
| `.docx` / `.doc` | 使用 docx 相关能力（pandoc / tencent-local-office-edit） | 确保段落与表格按文档真实顺序提取，正确处理自动编号列表。不得先汇总全部段落再汇总全部表格 |
| `.xlsx` / `.xls` | 使用 xlsx 相关能力 | 注意合并单元格、多 sheet、公式与显示值的区别 |
| `.pdf` | 使用 pdf 相关能力 | 扫描版/图片型 PDF 须先做 OCR，注意多栏排版可能导致顺序错乱。绘本 PDF 额外提取关键页画面描述 |
| `.json` | Read 工具直接读取 | 确认 JSON 结构与 schema 定义匹配 |
| `.png` / `.jpg` | Read 工具直接查看（多模态） | 可获取视觉内容描述，不可获取图层/矢量信息 |
| **嵌入的网页链接**（任何文件类型中提取出的 URL） | 文本提取后用 WebFetch 工具访问各 URL，获取网页内容 | 不得只复制链接到 wiki；必须访问后提取规则/方法论/数据等实质内容，结构化后写入对应 wiki 页面。网页内容与文件内容共同作为知识来源 |

- 专用工具因环境缺失不可用时，可临时用脚本兜底，但必须在摄入结果说明中明确注明"本次使用了非标准提取方式"。
- 任何**结构性结论**（如"某关键页面缺失""成长点与已有重复""设定冲突"等）写入 wiki 页面前，必须附带可复核的原始提取片段或引用位置作为依据，不得凭单次提取直接下结论。

## Web Link Processing Protocol（网页链接处理协议）

当 raw 文件的提取内容中包含网页链接（URL）时，按照本协议处理，**不得只把链接复制到 wiki 页面**。

### 扫描阶段

1. 在 `raw/` 文件文本提取完成后，扫描全部提取内容中的 `https?://` 等 URL 模式。
2. 去重并过滤：排除明显与内容无关的链接（如通用域名根链接），保留规则/政策/方法论/数据源等**可能包含实质性知识**的链接。

### 访问阶段

3. 使用 WebFetch 工具（或等效的网页内容提取能力）访问每个 URL，获取网页的实际内容。
4. 对同一来源域名下的多个分级页面（如 Common Sense Media 不同年龄段的评级指引），按需访问以获取完整数据集。
5. 如网页为 PDF 且 WebFetch 无法提取，尝试 Read 工具或脚本提取；若均不可行，在 `log.md` 中注明"二进制格式无法直接提取，保留参考"。

### 整合阶段

6. 将网页内容与原始文件内容合并，作为结构化 wiki 页面的共同知识来源。
7. **来源字段标注** `source` 须同时记录原始文件路径和访问的网页 URL：
   ```yaml
   source: "raw/xxx/行业规范.md + 各平台官方政策页面访问：YouTube / COPPA / FCC"
   ```
8. **版本字段递增**：同一页面因网页内容补充而更新时，递增 `version` 并更新 `last_updated`。

### 置信度判定

| 场景 | 置信度 | 说明 |
|---|---|---|
| 仅原始文件单一来源 | `unverified` | 默认，建议人工确认 |
| 网页内容确认了文件中的规则/数据但属同一机构来源 | `unverified`（维持） | 来源变多但权威性未跨级 |
| 网页内容提供了与文件内容完全不同的**独立知识维度**（如文件只列链接，网页提供完整政策原文） | `unverified`（维持，但页面质量已显著提升） | 在 `log.md` 记录为"网页补充" |
| 多个独立来源（文件 + ≥2 个不同机构的网页）得出相同结论 | `cross-validated`（可升级） | 在 `log.md` 注明依据 |
| 用户通过对网页内容的反馈确认了条目 | `feedback-confirmed` | 权重高于原始资料推断 |

### 输出要求

在 Completion（完成说明）中，除原定内容外还必须说明：
- 访问了哪些网页链接（URL 列表）
- 每个链接提取到了什么类型的知识（规则条文/方法论/数据/评级标准等）
- 哪些网页链接因技术原因未成功提取（二进制 PDF、需登录等），记录在 `log.md` 的备注中
- 网页内容导致的置信度变化

## Picturebook Source Routing（原始资料分流）

摄入原始资料时，根据 `raw/` 下用户放置的文件性质分流处理：

| 原始资料类型 | 判定特征 | 处理规则 | 写入目标 |
|---|---|---|---|
| 创作草稿（脚本） | 含 `**Text**` + `**插图描述**` 的 Markdown | 经人工确认后，按 content-spec 格式入库 | `wiki/projects/{project_id}/scripts/` |
| 创作草稿（创意） | 含故事梗概 + 核心文案 | 经人工确认后，补充到故事创意库 | `wiki/projects/{project_id}/`（增量更新） |
| 参考绘本 PDF | 真实绘本的 PDF 文件 | 提取元数据 + 标注关键页 + 生成索引 | 文件所在目录的 `index.md` |
| 参考图片 | 风格/角色/场景相关的图片 | 逐图编目 + 标签 + 生成索引 | 文件所在目录的 `index.md` |
| 视觉风格指南 | 含配色/造型/笔触规则的文档 | 整理为标准格式，检查覆盖维度 | 同目录（整理格式） |
| 用户反馈/修改意见 | 含具体修改点的文本 | 标注为"实战反馈"，权重高于模板自我推断 | 反哺 `wiki/log.md` 并标注相关页面 |
| 竞品分析 | 对竞品绘本的结构化分析 | 提取可复用方法论，经确认后入库 | `wiki/domains/`（方法论补充） |

## Domain Dedup & Merge（领域去重与合并）

摄入前（在 Read First 之后、正式提取之前）：

1. 读取 `wiki/index.md` 现有项目和领域列表。
2. 判断本次资料是否命中已有项目或领域：
   - 同一个 IP 的新剧集 → 归入已有项目（增量更新）
   - 不同 IP 的全新资料 → 新建项目目录
   - 通用方法论 → 归入 `wiki/domains/` 已有领域或新建
3. 如不确定归属，**必须先向用户确认**"归入已有项目（增量更新）还是新项目（新建目录）"，不得自行决定。
4. 用户确认新建才创建新目录；确认合并则走增量更新流程。

## Sync Workflow（全量同步——增删改全覆盖）

wiki-ingest-agent v4 新增 `sync` 模式，通过 `ingest-manifest.json` 追踪 raw/ → wiki/ 映射，自动检测并处理所有变化。

### ingest-manifest.json 结构

```json
{
  "_version": 1,
  "_last_ingest_at": "2026-07-08T15:00:00+08:00",
  "entries": [
    {
      "raw_path": "raw/xxx/doc.docx",
      "raw_hash": "sha256:abc123...",
      "raw_mtime": 1717740000.123,
      "wiki_pages": ["wiki/domains/xxx/page.md"],
      "ingested_at": "2026-07-08T14:30:00+08:00",
      "status": "active"
    }
  ]
}
```

字段：`raw_path`（唯一键）、`raw_hash`（SHA-256）、`wiki_pages`（数组）、`status`（active/deprecated）。

### 三步流程

```
Step 1: 扫描 raw/ → 获取当前文件树 + 每个文件的 SHA-256 hash
Step 2: 比对 manifest → 生成差异报告（新增/修改/删除/无变化）
Step 3: 按差异类型分路径处理
```

### 差异检测规则

| 条件 | 判定 |
|---|---|
| raw 有、manifest 无 | **新增** |
| raw 有、manifest 有、hash 不同 | **修改** |
| raw 有、manifest 有、hash 相同 | 跳过（无变化） |
| raw 无、manifest 有且 status=active | **删除** |

### 三路径处理

#### 🆕 新增
- 生成完整 wiki 页面
- 标注"首次摄入-单一来源"，置信度 `unverified`
- 追加到 `ingest-manifest.json`

#### ✏️ 修改
- 按 manifest 中的 `wiki_pages` 定位受影响的页面
- 重新提取 → 逐段比对 → 新增/修改/删除分别处理
- 原内容不丢失（移至「历史版本」折叠区或标注 `[已移除]`）
- `human-approved` 内容降级为 `cross-validated`
- 更新 manifest 中的 hash

#### 🗑️ 删除
- **不删除 wiki 内容**
- manifest 中标记 `status: "deprecated"`
- wiki 页面顶部插入废弃提示
- wiki/index.md 标注 `[已废弃]`

### 首次摄入（无 manifest）

manifest 不存在时，所有 raw 文件均视为新增。首次摄入完成后自动创建 manifest。

## Create vs Update Workflow（手动指定模式）

`ingest_type="create"` 或 `"update"` 时，仅处理用户指定的 `source_files`，不执行全量差异检测。

### 首次创建
- 生成完整的 wiki 页面（标题/摘要/来源/正文）
- 页面状态标注"首次摄入-单一来源"
- 更新 `wiki/index.md` 添加导航
- 写入 `wiki/log.md`

### 增量更新
- **不得整页覆盖重写**。必须读取现有页面，逐条比对新资料，做以下操作之一：
  1. **新增条目**：追加到对应章节，标注来源
  2. **修正已有条目**：保留修正前内容，以"更正记录"形式追加说明修正原因和新来源
  3. **确认已有条目**：提升其置信度等级
- `wiki/log.md` 必须体现 diff：新增了几条、修正了几条（含原因）、置信度升级了几条
- 示例（剧集创意库增量）：
  ```
  - 新增单集条目 N 条
  - 修正单集 X 的成长点描述（原描述与脚本正文不一致）
  - 确认单集 Y 的蓝思值预估（经 lexile-check 验证，置信度升级）
  ```

## Confidence Levels（置信度分级）

写入 wiki 的知识条目须标注置信度：

| 等级 | 含义 | 升级条件 |
|---|---|---|
| `unverified` | 单一来源，AI 主观归纳 | 默认初始等级 |
| `cross-validated` | ≥2 个独立来源观察一致 | AI 摄入时自动升级，在 `log.md` 注明依据 |
| `feedback-confirmed` | 来自用户反馈/实战修改意见 | 权重高于模板和原始资料推断 |
| `human-approved` | 经人工显式确认放行 | **AI 摄入流程不得自动打此标签**，仅在 Completion 中建议"人工确认后升级" |

页面头部状态取页面内条目最低档位；条目旁单独标注各自档位，避免笼统覆盖。

---

## Output Policy

- 未经确认的知识不直接写正式 `wiki/`，先标注待确认
- PDF/图片摄入：只产出索引文件至文件所在目录，不修改 `wiki/`
- 风格指南：整理为统一格式
- 所有摄入结果标注来源文件路径和置信度等级
- `raw/` 下文件只增不改，摄入时只读不写

## Completion

完成后必须说明：

- 本次模式：**sync（增删改全覆盖）** / **create（首次创建）** / **update（增量更新）**
- sync 模式时输出差异摘要：新增 N 文件、修改 M 文件、删除 K 文件、无变化 X 文件
- 处理了哪些原始资料（类型 + 数量 + 来源路径）
- 生成了哪些文件（含具体路径）
- 废弃了哪些 wiki 页面（如有）
- 是否需要人工确认（标注具体哪些内容待确认）
- 是否需要更新 `wiki/index.md` 和 `wiki/log.md`
- 涉及的置信度变化：哪些条目升级/降级/维持
- 是否有条目建议人工确认升级为 `human-approved`
- 本次提取是否使用了非标准提取方式（脚本兜底），如有需提醒人工复核结构性结论
- 对于 PDF/图片摄入：索引文件是否需要补充
